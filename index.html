What is this IP Address "169.254.169.254"?

I was going through the documentation of AWS when I found a totally new thing which says that an amazon instance may access instance-specific metadata as well as data supplied when launching the instance by making a query at http://169.254.169.254/. At first read, I felt how it is possible that all instances can get its own specific metadata by hitting the same URL.

So I logged into two of the instances and queried the same thing by randomly picking one of the metadata called user-data as given in the documentation. The result shocked me once again.


pradeep@nat-host-1b$ curl -s http://169.254.169.254/latest/user-data
nat-host-1b
pradeep@nat-host-1c$ curl -s http://169.254.169.254/latest/user-data
nat-host-1c

After doing some google search, I got RFC 5735 which says it to be one of the Link-Local IPv4 Address. I had heard of IPv6 Link-Local addresses but never of its kind in IPv4 address scheme.

Finally What is this heck ??

Suppose you have configured your eth0 to take IP from a DHCP server and by chance if your DHCP server is itself down, then it won't get any IP which will prevent both the communication within the LAN as well as to any machine outside the LAN ( or Internet). But for just LAN communication, whether getting an IP from DHCP is really important, can't we not just give IP from some subnet to all machine and start talking to each other. Yeah !!. This is the Hit. So what IETF has done is they have reserved one block (169.254.0.0/16 - Link Local Block) for this reason only. So if any interface configured to take IP through DHCP don't receive any response for its DHCP Discover Packets within the Timeout Period, it takes an IP (Candidate IP) from that Link Local Block. But before taking any IP its first confirms that no other device is having the same IP address which it has selected. So for this, it just do an ARP probe for that IP. If a reply is received to the ARP, it indicates the candidate IP address is already in use; a new random candidate IP address is then created and the process repeated. The process ends when there is no reply to the ARP, indicating the candidate IP address is available.
